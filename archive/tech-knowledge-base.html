01

Imaging & Camera

影像采集 / 计算摄影 / 专业格式

AVCaptureMultiCamSession

Apple 原生
多摄像头同时采集 · AVFoundation

在一个采集会话里同时调用前后/广角/长焦多颗摄像头，可同时输出多路预览、视频、照片或元数据，做画中画、双路录制、分屏等效果。

  • 能力前后同录、多路 VideoDataOutput / PhotoOutput / 元数据并行（条码、人脸）
  • 限制需 A12 及以上芯片；设备支持的组合由 supportedMultiCamDeviceSets 决定；功耗/发热较高
  • 调用AVFoundation，Swift/ObjC；通过 AVCaptureConnection 定义数据流向

Dual Photo Delivery

Apple 原生
双镜头同时出片 · AVCapturePhotoOutput

在双摄设备上同时拍下广角与长焦两张照片，可用于景深合成、视角对照或后期选择，无需用户来回切换镜头。

  • 能力一次快门同时返回两颗镜头的原片
  • 限制仅双摄设备；需在采集管线提前启用配置
  • 调用isDualCameraDualPhotoDeliveryEnabled

Semantic Segmentation Mattes

Apple 原生
人像语义分割遮罩 · 计算摄影

拍照时自动生成头发、皮肤、牙齿等语义分割遮罩，可在后期对各区域分别处理，实现精细抠像、美颜、影棚级人像效果。

  • 能力分通道输出头发/皮肤/牙齿等遮罩（matte）
  • 限制主要针对人像；依赖支持的设备与拍摄模式
  • 调用AVFoundation 照片管线（WWDC19 起）

ProRAW / Apple Log / ProRes

Apple 原生
专业影像格式 · 高宽容度采集

面向专业拍摄的高保真采集格式：ProRAW（计算摄影 + RAW 后期宽容度）、Apple Log（调色宽容度）、ProRes（高码率视频），适合需要重度后期的影像类 App。

  • 能力更大后期空间、保留计算摄影信息、专业调色流程
  • 限制文件体积大；机型/存储要求高；需对应 Pro 机型
  • 调用AVFoundation / 系统相机配置

Lens Smudge Detection

Apple 原生
镜头脏污 / 模糊检测 · Vision (iOS 26)

iOS 26 在 Vision 中新增镜头脏污检测，可在相册或自有采集管线中识别可能被指纹/脏污影响的模糊图片，提升扫描和拍摄质量。

  • 能力识别潜在脏污/模糊图像
  • 限制iOS 26+；属于辅助质量判断
  • 调用Vision framework（WWDC25）
02

On-Device Vision AI

本地视觉智能 / OCR / 文档结构识别

RecognizeTextRequest (OCR)

Apple 原生
图像文字识别 · Vision

从图片中识别印刷与手写文字，返回文本及位置框，全程本地运行。是扫描类、证件/票据识别、文档数字化的基础能力。

  • 能力多语言印刷+手写 OCR，带位置与置信度
  • 限制复杂排版需配合区域检测；语言覆盖有限
  • 调用Vision（iOS/macOS/iPadOS/visionOS），完全离线

RecognizeDocumentsRequest

Apple 原生
文档结构化识别 · Vision (iOS 26)

iOS 26 新增：不仅识别文字行，还能把文字组织成段落、读取表格的行列结构、识别列表与条码，保留文档的结构信息——非常适合智能扫描 App。

  • 能力段落分组、表格行列、列表、条码，支持约 26 种语言
  • 限制iOS 26+；结构识别受版面复杂度影响
  • 调用Vision framework（WWDC25）

Face / Landmark Detection

Apple 原生
人脸与关键点检测 · Vision

检测画面中的人脸、面部关键点与朝向，可用于自动对焦、贴纸/美颜、活体辅助、相册整理等。本地实时运行。

  • 能力人脸框、面部 landmark、姿态朝向
  • 限制非身份识别（不做"这是谁"）；遮挡/低光下精度下降
  • 调用Vision；可与采集管线实时联动

Object / Pose / Trajectory

Apple 原生
物体·人体·轨迹分析 · Vision

检测与追踪物体、人体与手部姿态、运动轨迹等。配合 Core ML 自定义模型可做运动分析、AR 交互、动作教学（如挥拍分析）。

  • 能力物体追踪、body/hand pose、trajectory 分析
  • 限制自定义识别需搭配训练好的 Core ML 模型
  • 调用Vision + Core ML，离线

Core ML

Apple 原生
端侧模型推理引擎 · 底座

Apple 的端侧模型运行框架，把训练好的视觉/音频/NLP 模型部署到 CPU/GPU/神经引擎(ANE)上推理。Vision 等高层框架底层都依赖它，也可直接跑自定义模型。

  • 能力自定义模型本地推理，自动调度 ANE/GPU/CPU
  • 限制需将模型转换为 .mlmodel/.mlpackage
  • 调用Core ML；常配合 Create ML 训练

Visual Intelligence

Apple 原生
系统级视觉理解 · Apple Intelligence

iOS 26 起可用于截图：圈选屏幕内容进行搜索、提问（接入 ChatGPT），并允许第三方 App 接入。属于系统级"识屏 + 问答"能力。

  • 能力圈选识别、屏幕内容搜索/提问
  • 限制需 Apple Intelligence 机型；部分能力联网
  • 调用系统集成 + App Intents 接入
03

Speech Recognition (ASR)

语音识别 / 语音转文字

OpenAI Whisper

开源
通用多语言转录基座 · 模型权重

OpenAI 开源的 ASR 模型家族（tiny→large-v3 / turbo），680k 小时训练，支持 99+ 语言。本身是模型权重而非成品 App，需配合封装使用。生态成熟，是大多数转录产品的底座。

  • 能力99+ 语言、自动语种识别、时间戳、标点
  • 限制原版依赖 Python/GPU 较重；非实时流式
  • 调用pip / MIT 许可；生态：faster-whisper、whisper.cpp、WhisperKit

whisper.cpp / WhisperKit

开源
端侧 Whisper 部署 · C++/CoreML

把 Whisper 移植到 C/C++，可在 CPU、Apple Silicon（CoreML）、甚至 WebAssembly 上高效运行。WhisperKit 进一步把它包成 Apple 平台原生的端侧转录方案，数据不出设备。

  • 能力离线转录，Apple Silicon 上性能好，隐私友好
  • 限制大模型对内存有要求；实时性取决于模型尺寸
  • 调用whisper.cpp（C/C++）、WhisperKit（CoreML/Swift）

Apple Speech Framework

Apple 原生
系统语音识别 · 端侧优先

Apple 自带的语音识别框架，可端侧运行（隐私好、可离线、实时），适合做听写、语音输入、语音命令。相比云端 Whisper，优势在隐私与实时响应。

  • 能力实时听写、端侧处理、与系统深度集成
  • 限制语种/词汇覆盖不如 Whisper 灵活
  • 调用Speech framework（SFSpeechRecognizer）

NVIDIA Parakeet

开源
超快流式英语 ASR · 实时

NVIDIA 的 RNN-Transducer 系列，主打推理速度与流式实时识别，RTFx 极高（处理速度远超 Whisper），适合需要低延迟实时转写的场景，准确率排名略低但够用。

  • 能力流式实时、极高吞吐、低延迟
  • 限制以英语为主；准确率非顶尖
  • 调用开源模型，Open ASR Leaderboard 可比

Mistral Voxtral Transcribe 2

开源 / API
高准确率 + 原生流式 · 新锐

Mistral 2026 年的二代 ASR，FLEURS 多语言基准上 WER 约 5.9%（优于 Whisper 的 ~7.4%），含离线批处理 Mini 版与因果编码器的实时流式版。语言覆盖较少（约 13 种），是值得关注的新选手。

  • 能力更低 WER、原生实时流式
  • 限制语言约 13 种，远少于 Whisper 的 99+
  • 调用Voxtral Mini（批处理）/ Voxtral Realtime（流式）

云端 ASR API

云端
托管转录服务 · 免运维

Deepgram Nova-3、Google Chirp、AssemblyAI、OpenAI GPT-4o-transcribe 等托管服务，主打开箱即用、规模化、实时流式与高级功能（说话人分离等），代价是数据上云与按量计费。

  • 能力免运维、可扩展、说话人分离、实时
  • 限制数据出设备、按量付费、需联网
  • 调用各家 REST/WebSocket API
04

Text-to-Speech (TTS)

语音合成 / 配音 / 语音克隆

Kokoro-82M

开源
超轻量端侧 TTS · 效率王

仅 8200 万参数、约 300MB，可在 CPU 上运行，质量却接近大很多倍的模型，2026 年初登顶 TTS Arena。适合本地、低算力、零成本、需要嵌入式部署的场景，但主打英语、不支持语音克隆。

  • 能力极快（GPU 上数十倍实时）、可 CPU 运行、免费、Apache 2.0
  • 限制英语为主、无克隆、情感表现弱于商用
  • 调用开源权重；可在 M 系 Mac via MPS 运行

Chatterbox / Chatterbox-Turbo

开源
高质量语音克隆 · ElevenLabs 级

Resemble AI 的开源模型（MIT），单 GPU 即可做语音克隆。厂商盲测中偏好率显著超过 ElevenLabs，是想自托管、追求高质量与克隆能力时的首选之一。

  • 能力单 GPU 语音克隆、质量对标/超过商用
  • 限制需 GPU；自托管要自己做工程
  • 调用开源权重，MIT 许可

ElevenLabs

云端
商用配音标杆 · 生态完善

商用 TTS 标杆，主打稳定、生态、海量语音库（4000+ 音色、70+ 语言）、即时克隆与 Studio 编辑器。卖的不只是音质，更是便利与可靠性。Flash 档延迟约 75–150ms。

  • 能力海量音色/语言、即时克隆、Studio、低延迟档
  • 限制按量付费、数据上云、需联网
  • 调用REST API；订阅制（约 $5/月起）

Fish Audio S2 Pro

开源 / API
多语言高质量 · 性价比

开源权重，双自回归（Dual-AR）+ RVQ 音频编解码设计，10M+ 小时多语言训练。托管 API 首字音约 100ms，价格约 $15/百万字符（远低于 ElevenLabs），多语言整体质量优秀。

  • 能力多语言、流式低延迟、克隆、价格友好
  • 限制自托管需工程投入
  • 调用开源 + 托管 API

VibeVoice / Dia2

开源
长篇 / 多人对话合成 · 播客级

面向长音频与多说话人对话的开源模型：VibeVoice 擅长长篇富表现力叙述（整集播客），Dia2 擅长多说话人对话。适合做有声书、播客、剧情配音。

  • 能力长篇连贯、多说话人、富表现力
  • 限制资源消耗较高；需 GPU
  • 调用开源权重

Cartesia Sonic / Qwen3-TTS

云端 / 开源
超低延迟实时 · 语音代理

主打极低延迟（Cartesia Sonic Turbo 约 40ms、Qwen3-TTS 约 97ms 流式），为实时语音对话/语音代理设计。在需要"边想边说"的交互场景里是关键技术。

  • 能力40–100ms 级延迟、原生流式
  • 限制顶级实时档多为云端/付费
  • 调用API / 部分开源

AVSpeechSynthesizer

Apple 原生
系统内置语音合成 · 零成本

Apple 系统自带 TTS，离线可用、零成本、与系统集成（含无障碍 Personal Voice 个人语音克隆）。音质和表现力不及顶级商用，但作为基础朗读/无障碍功能足够。

  • 能力离线、多语言系统音色、Personal Voice
  • 限制表现力/自然度有限
  • 调用AVFoundation（AVSpeechSynthesizer）
05

On-Device LLM

端侧大模型 / 文本智能

Apple Foundation Models

Apple 原生
系统端侧 LLM 框架 · iOS 26

iOS 26 起开放给第三方：直接调用 Apple Intelligence 核心的端侧 ~3B 模型（2-bit 量化），离线、免费、隐私好。擅长摘要、抽取、分类、改写、短对话——不是通用世界知识聊天机器人。最少三行 Swift 即可接入。

  • 能力摘要/抽取/分类/改写、Guided Generation 结构化输出、工具调用
  • 限制需 Apple Intelligence 机型；不擅长世界知识/复杂推理
  • 调用FoundationModels（LanguageModelSession），@Generable 宏

云端前沿 LLM API

云端
强推理 / 世界知识 · 服务器规模

当任务需要复杂推理、长上下文、世界知识或多模态理解时，端侧小模型力不从心，需调用 Claude、GPT 等服务器规模模型。代价是联网、按量计费、数据出设备。常与端侧模型分工：端侧做轻任务、云端做重任务。

  • 能力复杂推理、长上下文、多模态、世界知识
  • 限制联网、按量付费、隐私需评估
  • 调用各家 REST API（可在 App 内编排端云协同）

开源本地 LLM (Llama / Qwen / MLX)

开源
自托管端侧模型 · 可控

在设备/自有服务器上跑开源模型（Llama、Qwen 等），通过量化（GGUF）或 Apple 的 MLX 框架在 Apple Silicon 上推理。比系统模型更可控、可选更大尺寸，适合需要自定义、特定领域或更强能力又想离线的场景。

  • 能力尺寸/模型自选、可微调、离线可控
  • 限制工程与内存成本高；大模型端侧吃力
  • 调用MLX（Apple Silicon）、llama.cpp / GGUF 量化

技术组合灵感 · Stacks

知识库的真正价值在组合。下面是几条把上述能力拼起来、做出"不一样软件效果"的思路，供后续探索。

采集 → OCR → 结构化 → LLM

智能扫描 / 文档助手

镜头脏污检测保证图像质量 → RecognizeDocumentsRequest 读出表格/段落结构 → Foundation Models 本地摘要、提取字段、生成结构化数据。全程离线、隐私友好。

ASR → LLM → TTS

实时语音助手 / 口译

Parakeet/Voxtral 流式识别 → 云端或端侧 LLM 理解与生成 → Cartesia Sonic 超低延迟合成。延迟链路是体验关键，三段都要选低延迟方案。

多摄 → 分割 → 实时合成

双视角创作相机

MultiCamSession 前后同录 + 语义分割遮罩做实时人像抠像/背景替换，配 ProRes 专业格式输出，做 Vlog/采访类创作工具。

Whisper → 时间戳 → TTS 克隆

视频配音 / 译制

whisper.cpp 端侧转录带词级时间戳 → LLM 翻译 → Chatterbox 语音克隆按时间轴重新配音，做本地化译制工具。

Vision 姿态 → Core ML → LLM 反馈

动作教学 / 运动分析

Vision 提取身体/手部姿态与轨迹 → Core ML 自定义模型判定动作 → Foundation Models 生成针对性、口语化的改进建议（如挥拍/健身指导）。

端侧轻任务 ↔ 云端重任务

端云协同编排

用 Foundation Models 处理高频、隐私敏感、轻量的本地任务；遇到复杂推理/世界知识再路由到云端 LLM。在 App 内做智能编排，兼顾成本、隐私与能力。
